## 一句话总结

EO-1 用单个 3B 参数、初始化自 Qwen2.5-VL 的 decoder-only Transformer,在同一因果序列里混合"离散自回归解码文本"与"连续 flow matching 去噪动作"两种目标,并在自建的 150 万条交错视觉-文本-动作数据集 EO-Data1.5M 上训练,使推理与控制沿时间轴交替、互相提供上下文;结果在 LIBERO(98.2% 成功率)、SimplerEnv、自建 EO-Bench 以及 Franka/WidowX/AgiBot G-1/SO100 四种真机平台共 28 项任务上全面超过 π0、GR00T-N1.5、OpenVLA-OFT 等开源基线,且推理端只需单张 RTX 4090、6GB 显存即可实时运行。

## 一、问题与动机

生成式通用机器人策略要走向开放世界,核心瓶颈是"开放世界泛化"——机器人需要具备完整的世界知识、类人的推理能力以及灵巧的执行能力。早期 VLA 工作(OpenVLA、π0、π0-FAST 等)把 VLM 扩展成动作模型,要么通过离散动作 token 做自回归解码,要么额外接一个连续 flow matching/diffusion 动作专家模块;但由于这些模型仅在机器人数据上做微调或对动作模块单独训练,导致继承自 VLM 的通用语义知识被削弱,指令跟随能力受限,只能覆盖狭窄的任务域。

近期的协同训练(co-training)工作(如 π0.5、knowledge-insulating VLA)开始把网络文本-图像数据和机器人数据混合训练,在面对新物体、新背景时展现出更好的泛化性。但论文指出这些方法有一个共同缺陷:它们几乎都是把机器人动作放在整个 VLA 输出序列的**末端**生成,忽视了视觉、语言、动作三种模态之间本应存在的丰富时间动态与因果依赖——人类在操作物体时,推理会指导动作,动作结果又会反过来更新推理("reasoning guides action and action results inform subsequent reasoning")。

由此,论文提出核心研究问题:**如何设计一种有效的训练范式,让通用机器人策略支持灵活、双向互通的"推理-动作"整合?** 为此需要同时具备(i)一个能够无差别处理文本、图像、视频、动作等多模态输入/输出的统一架构,以及(ii)一个把文本、图像、视频和机器人动作有机组织在一起的、结构化的大规模多模态具身数据集。

## 二、核心方法

EO-Robotics 工具链由三部分组成:**EO-1** 模型、**EO-Data1.5M** 数据集、**EO-Bench** 评测基准。

### 2.1 问题建模

把通用机器人策略建模为一个统一模型 $\pi_\theta$,联合刻画多模态具身推理与机器人控制:

$$\pi_\theta(\hat{\mathbf{a}}_{t:t+h}, \hat{\ell}_t \mid \mathbf{o}_t, \ell_t, \mathbf{a}_{t-h:t})$$

其中 $\mathbf{o}_t = [\mathbf{I}_t^1,\dots,\mathbf{I}_t^n, \mathbf{q}_t]$ 是多视角图像观测与机器人状态,$\ell_t$ 是具身推理文本(问答或任务指令),$\hat{\mathbf{a}}_{t:t+h}$ 是长度为 $h$ 的动作 chunk。

**用大白话说**：这一个式子把"下一步该怎么想"(文本 $\hat\ell_t$)和"下一步该怎么做"(动作 $\hat{\mathbf a}_{t:t+h}$)放进同一个条件分布里,模型看到的是过去的图像、状态、文本和动作,一次性决定接下来说什么、做什么,而不是像传统 VLA 先把话说完再统一去生成动作。

整个交错 token 序列 $x_{1:N}$ 的自回归展开写成 $p(\hat x_{1:N}) = \prod_{i=1}^N p(\hat x_i \mid x_{<i})$。每个 token 可以是离散文本 token、连续图像 patch token、连续机器人状态 token,或部分去噪的动作 token;文本走语言建模头(logits + 交叉熵),动作走 flow matching 头。视觉与机器人状态 token 只作为输入,不参与输出预测。

### 2.2 模型架构:统一 decoder-only + 双头

EO-1 的骨干直接初始化自 **Qwen2.5-VL**(3B),文本 tokenizer 和视觉编码器沿用预训练 VLM,机器人状态 $\mathbf q_t$ 用一个随机初始化的线性投影层映射进同一 embedding 空间。动作生成走 **rectified flow**(Lipman et al., 2022):"带噪动作" $\mathbf a_t^\tau = \tau \mathbf a_t + (1-\tau) z_\tau$($z_\tau\sim\mathcal N(0,I)$)经另一层随机初始化的线性投影器嵌入为噪声动作 token,再由 Transformer 的输出经 flow head 映射出去噪向量场,按前向欧拉积分从 $\tau=0$(纯噪声)积分到 $\tau=1$(干净动作):

$$\hat{\mathbf a}_t^{\tau+\delta} = \hat{\mathbf a}_t^\tau + \delta\, V_{\pi_\theta}\big(\hat{\mathbf a}_t^\tau \mid \mathbf o_t, \ell_t, \mathbf a_{t-h:t}\big)$$

**用大白话说**：动作生成不再是"挤牙膏式"逐 token 采样离散动作,而是先扔一坨随机噪声进去,让模型学会的"速度场" $V_{\pi_\theta}$ 一步步把它"吹"成真实的连续动作轨迹,10 步左右积分就能出一个 16 步动作 chunk——比自回归离散化动作快得多、也更连续平滑。

关键设计是**不引入任何额外的动作专用大模块**(不像 π0、Driess et al. 那样接一个独立的 action expert),只加了两个小 MLP(状态投影、噪声动作投影),从而让动作生成与文本生成共享绝大部分 Transformer 参数,便于跨模态知识从 VLM 迁移到动作。

训练目标是自回归文本损失与 flow matching 损失之和 $\mathcal L=\mathcal L_{ar}(\theta)+\mathcal L_{fm}(\theta)$,其中

$$\mathcal L_{fm}(\theta)=\mathbb E_\tau\Big[\big\|V_\theta(\mathbf a_t^\tau,\tau\mid x_{<a})-(z_0-\mathbf a_t)\big\|^2\Big]$$

$\tau$ 从一个偏向低(更嘈杂)时间步的 Beta 分布中采样(沿用 Black et al., 2024 的做法)。

### 2.3 交错格式与 Interleaved Rectifying Sampling

训练数据被组织成三种类型:多模态理解数据、机器人动作生成数据、混合模态生成数据(交错数据)。交错序列的通用格式为 `[BOI]{图像}[EOI][BOS]{文本}[EOS][BOR]{状态}[EOR][BOA]{噪声动作}[EOA][BOI]{图像}...`,训练时用全向(omnidirectional)因果注意力 mask 处理这三类数据,并缓存 KV 以加速推理时的混合模态生成。

一个训练难点是:交错序列里,动作生成段的"带噪动作 token"不应该被后续的文本/图像 token 关注(它们应该关注干净动作,而不是采样过程中的噪声中间态),否则会破坏多模态 token 之间本应有的因果关系。论文为此提出 **interleaved rectifying sampling**:对一条包含 $N$ 个动作生成段的交错序列采样 $N{+}1$ 个训练子序列——子序列若在第一个动作段结尾截断则不作处理;若子序列内部还包含更早的动作生成段(即该动作段并非结尾),就把该段内的带噪动作 token 替换为**干净动作 token**。这样每个动作生成段本身仍然用带噪 token 做 flow matching 去噪监督,同时又能通过干净动作 token 间接把梯度回传给后续的交错生成,兼顾了"去噪训练"和"因果一致性"。

### 2.4 EO-Data1.5M:交错具身数据构建

预训练语料分三大类(见下表,数字取自论文 Table 1/6):

| 模态 | 主要来源 | 样本数 | Token 数 |
|---|---|---|---|
| Web 多模态数据 | LLaVA-1.5、LLaVA-Video-178K、PixMo-Points、RefCOCO、RoboVQA | 5.7M | 7.1B |
| 机器人控制数据 | AgiBotWorld、Open X-Embodiment、RoboMIND、SO100-Community、IPEC-Franka | 1.2M episode | 127.3B |
| 交错具身数据 EO-Data1.5M | 从 AgiBotWorld / Open X-Embodiment / RoboMIND 中筛选标注 | 1.5M | 1.0B |
| **总计** | | **260.6M** | **135.4B** |

EO-Data1.5M 的构建流程(四步):

1. **机器人视频筛选与整理**：从 210 万条机器人 episode 出发,用预训练视觉骨干提取特征并做 K-means 聚类,人工核验去冗余,保证场景/任务多样性;
2. **视频切分与字幕**：人工标注员 + VLM 把长视频切成单个子任务的短片段,并生成描述性字幕;
3. **QA 对生成**：用 VLM 按设计好的模板生成"时间推理"(任务规划、物理常识)与"空间推理"(轨迹预测、物体指代、物体定位、多视角对应)两类问答,人工从多候选答案里选择/改写正确答案;
4. **清洗与改写**：规则过滤无效/歧义答案,再用 LLM 改写问答对提升文本多样性、注入模糊表述。

标注得到:**具身时间推理数据**(物理常识 7 万条 QA;任务规划/情节字幕/可供性 QA/过程验证/子任务 QA/失败检测共 70 万条 QA)与**具身空间推理数据**(关系推理、轨迹预测、物体指代、物体定位、多视角定位共 150 万条 QA)。再通过三种灵活格式——交错时间推理格式、交错空间推理格式(引入"轨迹指令"衔接轨迹预测 QA 与机器人动作)、交错自由对话格式——把推理 QA 与机器人控制数据沿时间轴拼接,最终得到 **12.2 万条**用于混合模态生成训练的交错视觉-文本-动作样本。

### 2.5 EO-Bench 评测基准

为解决现有具身推理评测"单题混杂多种推理能力、难以归因"的问题,论文自建 **EO-Bench**,按空间理解、物理常识、任务推理、状态估计四大类组织(各类下设子类,如轨迹推理、视觉定位、关系推理、多视角定位/直接影响与反事实推理/任务规划、情节字幕、过程验证/物体状态识别等)。论文正文对总题量的两处表述不完全一致:3.3 节写"648 道 QA(370 空间理解 + 140 任务推理 + 84 物理动态推理 + 48 物理常识)",4.2 节又写"700 道多选 VQA、四类均衡分布",此处如实记录原文两种说法供读者留意。

**统一推理-控制基准(4 任务,与"外接 GPT-4o 做规划 + GR00T-N1.5/π0 做控制"的分层基线对比)**：Tic-Tac-Toe 76% vs 24%/36%,视觉重排 79% vs 47%/66%,做三明治 84% vs 70%/67%,烤牛排 55% vs 41%/46%。单一交错策略在需要"边想边做"的任务上相对分层 pipeline 优势最大(Tic-Tac-Toe 领先 40 个百分点),论文将其归因于统一骨干消除了"规划器-控制器"接口处的误差传播。

**架构与数据消融(4.9 节)**：(i)混合解码(AR+flow)全面优于纯自回归 FAST-tokenizer 方案——LIBERO 上 EO-1(base) 98.2% vs EO-1(fast) 88.0%,且随演示数据从 50 条增至 5 万条,纯自回归的语言泛化精度停滞在 0.43,动作泛化停滞在 0.23,而混合解码持续上升;(ii)交错视觉-文本-动作数据能显著提升语言泛化(EO-1(interleaved) 从 0.27→0.82,EO-1(base) 仅 0.01→0.59);(iii)但**与任务不对齐的通用多模态指令数据反而有害**——引入 LLaVA-Instruct-150K/RefCOCO 的 EO-1(llava) 变体,在 Agibot Sandwich 上从 0.77 掉到 0.66,在 WidowX 泛化上从 0.71 骤降到 0.34,论文将其解释为"语言先验偏置削弱了物理接地(physical grounding)"。

**工程指标**：训练用 1.2M 真机 episode + 5.7M 网络多模态样本 + 1.5M 交错样本,共 135B token,5 epoch,DeepSpeed ZeRO-1;推理时以 10 步积分预测 16 步动作 chunk,单张 **NVIDIA RTX 4090、仅需 6GB 显存**即可实时运行。

## 四、局限性

论文在结论部分明确列出三点未来工作方向,也即当前版本的局限:

1. **场景覆盖仍不完整**：尚未很好地覆盖导航、避障、失败检测与分析、人类意图识别、人机协作等更复杂的开放世界场景,需要把更通用的多模态理解、具身推理与动作能力进一步整合进同一系统。
2. **架构/推理效率仍有提升空间**：论文承认当前"交错生成"本质上仍是顺序推进的因果解码,并非真正的类人"同步"推理与动作;作者计划探索更高效的统一架构设计或异步推理管线来实现更接近人类的并发推理-动作。
3. **数据覆盖的具身类型有限**：尚未纳入人类操作数据和更多人形机器人具身,计划扩充数据与具身类型使 EO-1 成为更广泛的通用基础模型。

此外,从实验部分可以读出两个隐含局限:(a)"域外动作泛化"仍主要依赖大规模、多样化的机器人演示数据本身——在演示数据有限的 Agibot Sandwich 任务上,交错推理数据并未带来提升(In-domain 与 interleaved 都停在 0.75 左右),只有像 WidowX 这种拥有大规模 BridgeData 支持的场景才能看到交错训练明显拉高泛化(0.71→0.80);(b)EO-Bench 是作者自建、自评的基准,且论文正文对其题量的两处描述(648 vs 700)不完全一致,第三方复现和跨论文横向比较的可靠性有待观察。

## 五、评价与展望

**优点**：EO-1 的核心贡献是把"统一多模态生成"(如 Show-o、Transfusion、JanusFlow 等图文统一生成模型的思路)第一次系统性地搬到了 VLA 领域,并做对了两件此前工作没有做到位的事——一是真正做到"零额外动作专家模块"的参数共享(相比 π0 系需要独立 action expert、HybridVLA 等需要额外融合机制),二是显式构造了"交错"的推理-动作训练信号(而非简单把 QA 数据和动作数据分开co-train,或把动作永远放在序列末端)。EO-Bench 数据集消融部分给出的证据也比较有说服力:同样是多模态数据,"交错对齐"的数据带来正向泛化增益,而"通用但不对齐"的 LLaVA 式指令数据反而会损害物理接地——这是一个对后续数据配方设计有参考价值的实证结论。工程上单卡 4090、6GB 显存即可实时推理,对社区复现和边缘部署也比较友好。

**与其他公开工作的关系**：EO-1 与 $\pi_0$/$\pi_{0.5}$、GR00T-N1.5、OpenVLA-OFT 同属"VLM 骨干 + 连续动作头"这一大类 VLA,区别在于后者普遍保留独立的 action expert/流匹配子网络(π0、GR00T)或坚持纯离散自回归(OpenVLA-OFT 用 FAST tokenizer),而 EO-1 选择让动作头和语言头共享同一套 Transformer 参数,这与 Show-o、Transfusion、JanusFlow 等"统一理解-生成"图文模型的技术路线一脉相承,只是把生成目标从图像换成了连续机器人动作。相较 ChatVLA、Magma 等同样强调"推理增强"的 VLA,EO-1 的差异化卖点是"交错"而非"先想后做"或"先做后想"的顺序拼接。

**开放问题与可能的改进方向**：(1)交错格式目前依赖人工设计的三种拼接模板(时间推理/空间推理/自由对话),这类模板本质上还是启发式的,是否存在更自动、更端到端学出"何时该推理、何时该动作"的机制,是一个开放问题;(2)flow matching 动作头与 LM 头共享骨干虽然带来了知识迁移优势,但训练目标性质迥异(去噪回归 vs 分类),论文并未深入分析两个损失在梯度层面的相互干扰(negative transfer)是否存在及其规模效应,只给出了固定权重 1.0 的经验设置;(3)EO-Bench 尚缺乏第三方复现和跨模型的独立验证,其"648/700"题量的内部不一致也提示评测细节有待进一步公开核实;(4)真正的"异步"或"系统2式"推理-动作解耦(作者在结论中也提到)仍是未解决的方向,与近期强调显式规划/慢思考的具身模型(如作者同组的 Hume)可能存在互补空间。
